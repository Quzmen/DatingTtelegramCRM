# UX/UI аудит и переработка Telegram CRM

Аудит сделан по коду (`frontend/`, `backend/`), скриншоты в задании не пришли — приложился только архив проекта. Ниже: что реально нашлось при разборе кода, а не гипотезы по описанию.

Важно сразу: приложение технически более зрелое, чем описано в ТЗ. Тут уже есть полноценный мессенджер-вид (чат, папки, медиатека, пересылка), карточка контакта с историей, дашборд с воронкой — это не «внутренняя админка», а близкий к продукту интерфейс. Проблемы, которые нашлись, точечные, а не «всё плохо».

---

## Этап 1. UX-аудит: проблема → почему плохо → решение

### 1. Отдельная страница Kanban дублирует CRM
**Проблема.** «Канбан» и «Контакты» — два независимых пункта меню, оба показывают тех же контактов, только в разной раскладке; между ними нет связи, кроме перехода по клику на карточку.
**Почему плохо.** Пользователь должен помнить, что тот же контакт есть в двух местах; это создаёт ощущение, что в системе два разных модуля вместо одного CRM. Лишний пункт в навигации, который ничего не добавляет к «Контактам».
**Решение.** Объединить в один раздел «CRM» с переключателем «Таблица / Канбан» — сделано (см. Этап 3).

### 2. Telegram-страница почти пустая
**Проблема.** До правок раздел «Telegram» состоял из формы входа и списка контактов для импорта, ограниченных по ширине 640px в центре экрана — при этом справа и снизу пустое пространство.
**Почему плохо.** У пользователя нет ответа на вопрос «а что вообще происходит с интеграцией» — сколько контактов уже импортировано, когда последний раз обновлялись данные, что делать дальше. Раздел выглядит как форма настройки, а не как рабочий экран.
**Решение.** Добавлена строка статистики (статус подключения, контактов в Telegram, импортировано, всего в CRM) и блок быстрых действий (импортировать новые, открыть CRM, создать кампанию, синхронизировать). Макет растянут на всю ширину.

### 3. У «Telegram» и «Кампаний» не было отступов от края экрана
**Проблема.** В CSS отступы (`padding`) для контента были явно заданы только для `#view-dashboard`. У `#view-telegram` и `#view-campaigns` их не было вовсе — контент упирался в край окна.
**Почему плохо.** Это прямая причина ощущения «интерфейс незавершён»: соседние разделы визуально ведут себя по-разному без причины.
**Решение.** Общий паддинг применён ко всем трём разделам (дашборд, Telegram, кампании) — сделано.

### 4. Пустое состояние «Кампаний» ничего не объясняет
**Проблема.** Если кампаний нет, показывалась одна строка текста: «Кампаний пока нет — создайте первую».
**Почему плохо.** Новый пользователь не понимает, что вообще такое «кампания» в этой системе, какие сценарии она покрывает, и с чего начать, кроме как открыть пустую форму.
**Решение.** Добавлено объяснение назначения раздела и 4 карточки-шаблона (приветствие, повторное касание, массовая рассылка, напоминание клиентам), которые сразу открывают форму создания с предзаполненными названием и текстом — сделано.

### 5. У списка кампаний нет фильтрации по статусу
**Проблема.** Все кампании, независимо от статуса (черновик/запущена/завершена/с ошибками), показывались одним нефильтруемым списком.
**Почему плохо.** При росте числа кампаний найти «что сейчас реально выполняется» или «где были ошибки» становится неудобно — приходится читать бейджи на каждой карточке.
**Решение.** Добавлены вкладки-фильтры с счётчиками: Все / Активные / Запланированные / Остановленные / Завершённые / С ошибками — сделано.

### 6. Дашборд и раздел «Контакты» пересекаются по смыслу
**Наблюдение, не тронуто.** Дашборд уже сфокусирован на сводке (счётчики, воронка, «требует внимания») и не дублирует список контактов — раздел не является лишним, менять не требуется.

### 7. Раздел «Telegram» показывает мнимые метрики, если их нет данных
**Осознанно не сделано.** В ТЗ просили показать «текущие лимиты Telegram» и «состояние API». В API проекта этих данных нет (backend их не отдаёт), и добавлять их бэкендом — значит менять бизнес-логику, что прямо запрещено в ограничениях задания. Показывать выдуманные цифры хуже, чем не показывать ничего — это подорвёт доверие к остальной статистике. Решение: показаны только реальные метрики (статус, счётчики контактов); если понадобятся лимиты API — это отдельная бэкенд-задача (добавить эндпоинт), не переработка интерфейса.

## Этап 2. Новая структура

**Навигация (левый рельс):** Дашборд → Чат → CRM → Telegram → Кампании. Канбан убран как отдельный пункт (перенесён внутрь CRM). Отдельного пункта «AI Аналитика» в проекте нет — в коде нет такого модуля; добавлять в меню раздел без реализации хуже, чем не добавлять (это создало бы ещё одно «пустое место»). Если AI-аналитика планируется, это отдельная задача на бэкенд + фронт, а не переименование меню.

**CRM (бывшие «Контакты» + «Канбан»):** один раздел с переключателем **Таблица | Канбан** в шапке.
- Таблица: список слева (поиск, фильтры по статусу/тегу/интересу) + карточка контакта справа — без изменений, уже работало хорошо.
- Канбан: то же самое, воронка по этапам с drag-and-drop — теперь это режим отображения тех же данных, а не отдельный инструмент.
- Клик по карточке в любом режиме открывает контакт в режиме «Таблица».

**Telegram:** экран интеграции, а не просто форма входа.
- Строка статистики сверху (4 карточки).
- Строка быстрых действий (импорт, открыть CRM, создать кампанию, синхронизировать).
- Ниже — сама форма подключения аккаунта и список контактов на импорт, как раньше, но на всю ширину.

**Кампании:** без изменений в структуре модалок и логике отправки (бэкенд не тронут), но:
- пустое состояние объясняет назначение раздела и предлагает 4 готовых шаблона;
- список кампаний теперь фильтруется по статусу вкладками со счётчиками.

## Этап 3. Реализация — что изменено в коде

Изменения только во фронтенде (`frontend/`), backend и API не тронуты — контракты те же.

1. **`frontend/index.html`**
   - Убрана кнопка «Канбан» из левого меню.
   - Раздел `#view-contacts` переструктурирован: добавлена шапка с переключателем «Таблица / Канбан» (`.seg-toggle`), контент разбит на `#contactsBodyTable` (старые список+карточка) и `#contactsBodyKanban` (канбан-доска, бывший `#view-kanban`).
   - Раздел `#view-telegram` дополнен контейнерами `#tgStatGrid` и `#tgQuickActions`.

2. **`frontend/js/app.js`**
   - `views` больше не содержит `"kanban"` как отдельный экран.
   - Добавлено состояние `contactsMode` и функция `setContactsMode()`, переключающая видимость таблицы/канбана внутри CRM и вызывающая `Kanban.render()` по требованию.
   - `App.goToContact()` теперь всегда возвращает CRM в режим «Таблица» — иначе при переходе из дашборда в контакт при открытом канбане карточка контакта была бы не видна.

3. **`frontend/js/contacts.js`**
   - `Contacts.goToContact()` — тот же фикс режима, что и выше (переход из карточки в канбане на карточку контакта корректно показывает детальную панель).

4. **`frontend/js/telegram.js`**
   - Добавлена `renderStatsAndActions()` — считает реальные метрики из уже загруженных данных (без новых запросов к API) и рисует 4 карточки статистики + 4 кнопки быстрых действий.
   - Вызывается после каждой загрузки контактов и при неавторизованном состоянии (тогда блоки скрываются, чтобы не показывать нулевую статистику до входа).

5. **`frontend/js/campaigns.js`**
   - Добавлены константы `CAMPAIGN_TEMPLATES` (4 шаблона) и `LIST_FILTERS` (6 вкладок фильтра).
   - `renderList()` переписан: пустое состояние теперь с иллюстрацией, объяснением и карточками шаблонов; непустой список — с вкладками-фильтрами над сеткой карточек.
   - `openEditModal()` принимает необязательный `template` и предзаполняет название/текст кампании при создании из шаблона.

6. **CSS (`style.css`, `campaigns.css`)**
   - Общий паддинг для дашборда/Telegram/кампаний.
   - Новые стили: `.seg-toggle` (переключатель режимов), `.contacts-body` (раскладка CRM), `.tg-quick-actions`/`.tg-quick-action` (кнопки быстрых действий), `.campaign-filter-tabs`, `.campaign-empty`, `.campaign-templates`.
   - Раскладка Telegram расширена на всю ширину (было `max-width: 640px`).

### Что не сделано в этой итерации и почему
- **AI Аналитика** как пункт меню — не добавлена, т.к. в проекте нет такого модуля ни на бэкенде, ни на фронте; добавление пустого пункта меню создало бы ту же проблему «незаполненности», с которой боролись.
- **Лимиты Telegram API / состояние API-подключения** на странице Telegram — не показаны, т.к. бэкенд не отдаёт эти данные, а трогать бэкенд запрещено условиями задания.
- **Полный визуальный рестайлинг всех экранов** (чат, карточка контакта, медиатека) — не входит в эту итерацию: дизайн-система (`style.css`, токены, карточки, тени) уже соответствует ориентирам (Linear/Notion — нейтральная палитра, единые радиусы и тени), новые блоки сделаны в тех же токенах. Если нужно менять сам визуальный язык (цвета, шрифты, плотность) — это отдельная, более рискованная итерация, которую лучше делать поэкранно с проверкой на реальных данных.
