# Деплой на Render

## Что было изменено для деплоя

- **Dockerfile** — слушает `$PORT` (Render сам его подставляет), а не
  захардкоженные `127.0.0.1:8730` из `run.py` (тот остаётся для локального
  запуска).
- **Basic Auth** (`backend/main.py`) — весь сайт закрыт логином/паролем.
  Список пар задаётся одной переменной окружения `APP_USERS`, можно
  добавить по паре на каждого друга: `me:pass1,friend1:pass2,friend2:pass3`.
  Без этой переменной приложение открыто всем, у кого есть ссылка —
  для CRM с личной перепиской это не годится.
- **`.gitignore`** — папка `data/` (база, сессия Telegram, кэш медиа)
  и `.env` полностью исключены из git. Сессия Telegram — это фактически
  ключ входа в ваш аккаунт, ей нельзя оказаться в репозитории, даже
  приватном.
- **`/api/admin/import-data`** — служебный эндпоинт, чтобы один раз
  перенести существующие `data/*` на сервер, минуя git.
- **`render.yaml`** — Blueprint-конфиг с постоянным диском на `/app/data`
  (без диска все данные будут теряться при каждом редеплое).

## Шаг 1. Git-репозиторий

```bash
cd telegram-crm
git init
git add .
git status   # убедитесь, что data/ и .env НЕ попали в список
git commit -m "Подготовка к деплою на Render"
git remote add origin <ссылка на ваш GitHub-репозиторий>
git push -u origin main
```

## Шаг 2. Создать сервис на Render

Вариант А — через Blueprint (проще, использует `render.yaml`):
1. Render Dashboard → New → Blueprint → выбрать этот репозиторий.
2. Render сам подхватит `render.yaml`, план `starter` и диск на 5 ГБ.

Вариант Б — вручную:
1. New → Web Service → выбрать репозиторий.
2. Runtime: **Docker**.
3. План — любой платный (не Free — на нём нет постоянных дисков).
4. Environment → Add Disk: mount path `/app/data`, размер под ваши нужды
   (медиа-кэш сейчас ~72 МБ и будет расти).

## Шаг 3. Переменные окружения

В Render Dashboard → Environment добавьте:

- `APP_USERS` = `login1:pass1,login2:pass2,...` — по паре на каждого,
  кому нужен доступ (обязательно, иначе сайт открыт всем).
- `AI_PROVIDER` = `local` (или `gemini`, если используете Contact
  Intelligence через Gemini).
- `GEMINI_API_KEY` — если `AI_PROVIDER=gemini`.
- `TG_API_ID` / `TG_API_HASH` — необязательно, в коде уже есть значения
  по умолчанию; задавайте, только если хотите использовать свои.

Нажмите Deploy.

## Шаг 4. Перенос текущих данных (сессия Telegram, база, медиа)

После первого успешного деплоя перенесите локальную папку `data/` на сервер,
не коммитя её в git:

```bash
cd telegram-crm/data
zip -r ../data_backup.zip . -x "upload_tmp/*"
cd ..
curl -u <login>:<пароль из APP_USERS> \
     -F "file=@data_backup.zip" \
     https://<ваш-сервис>.onrender.com/api/admin/import-data
```

Должен вернуться `{"status": "ok", ...}`. После этого можно удалить
`data_backup.zip` локально — секреты в нём такие же чувствительные, как
в исходной папке `data/`.

Откройте `https://<ваш-сервис>.onrender.com`, введите логин/пароль из
`APP_USERS` — приложение должно сразу показать, что Telegram уже
авторизован (заново вводить код не нужно).

Если предпочитаете не переносить старую сессию, а войти в Telegram заново —
просто пропустите этот шаг и авторизуйтесь через форму входа на самом сайте.

## После деплоя

- **Удалить/закрыть `/api/admin/import-data`**, если он больше не нужен
  (можно закомментировать `app.include_router(admin.router)` в
  `backend/main.py` и передеплоить) — он работает под тем же Basic Auth,
  но лишний открытый write-эндпоинт лучше не держать постоянно.
- Если когда-либо заподозрите, что сессия Telegram могла утечь —
  завершите её в самом Telegram: Настройки → Устройства → нужная сессия
  → «Завершить».
