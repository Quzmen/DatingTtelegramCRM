# Telegram Contacts CRM

Локальная CRM-панель для управления личными знакомствами из Telegram.
Работает полностью на вашем компьютере: без сервера в интернете, без
облака. Все данные хранятся в одном файле SQLite на диске.

AI-анализ переписки (см. «Contact Intelligence» ниже) по умолчанию тоже
считается полностью локально, без сети. По желанию его можно дополнить
реальным LLM-анализом через Google Gemini API (свой ключ) — это
единственное место в проекте, которое обращается в интернет, и оно
строго опционально.

## Стек

- **Backend:** Python + FastAPI + SQLAlchemy
- **База данных:** SQLite (`data/crm.db`)
- **Frontend:** HTML / CSS / vanilla JavaScript (без сборки, без npm)

## Запуск (один раз — установка)

Нужен Python 3.10+.

```bash
cd telegram-crm
pip install -r requirements.txt
```

## Запуск приложения (каждый раз)

```bash
python run.py
```

Откроется браузер на `http://127.0.0.1:8730`. Если не открылся сам —
перейдите по ссылке вручную. Чтобы остановить сервер — `Ctrl+C` в
терминале.

При первом запуске автоматически создаётся файл базы данных
`data/crm.db` со всеми нужными таблицами. Никакой ручной настройки
не требуется.

## Возможности

- **Дашборд** — количество контактов, новые за неделю, активные
  диалоги, контакты, требующие внимания, разбивка по этапам воронки.
- **Карточки контактов** — имя, username, фото (по ссылке), источник
  знакомства, статус, уровень интереса (1–10), заметки, теги,
  следующая задача, дата последнего контакта.
- **Канбан-доска** — 6 колонок (Новый → Тёплый → В работе → Назначена
  встреча → Встречались → Архив), карточки перетаскиваются мышкой.
- **История взаимодействия** — хронология заметок, сообщений и
  встреч по каждому контакту, с возможностью добавлять и удалять
  записи.
- **Напоминания** — блок «Требуют внимания» на дашборде показывает,
  с кем не было контакта дольше 7 дней.
- **Поиск и фильтры** — по имени/username, статусу, тегу, уровню
  интереса.
- **Contact Intelligence (AI-анализ)** — оценка интереса контакта
  (0–100) по поведению в переписке, предлагаемый статус, следующее
  действие и краткое summary; по желанию — с реальным LLM-анализом
  и черновиком ответа (см. раздел ниже).

## Contact Intelligence (AI-анализ)

Кнопка обновления анализа на карточке контакта запускает подсчёт
`interest_score`, категории, предлагаемого статуса и следующего
действия. Это работает в двух режимах, которые переключаются одной
переменной `AI_PROVIDER`:

- **`local` (по умолчанию)** — всё считается локальной эвристикой
  (`backend/analysis.py`) прямо по истории переписки: кто первый
  пишет, скорость и длина ответов, вопросы, инициатива встречи и т.д.
  Без сети, без ключей, без сторонних сервисов.
- **`gemini`** — поверх той же самой эвристики (она по-прежнему
  считает `interest_score` и статус) подключается Google Gemini API,
  который переписывает summary и next_action живее и конкретнее, а
  также предлагает черновик следующего сообщения.

В режиме `gemini` числовая оценка интереса и правило выбора статуса
всё равно остаются полностью локальными — Gemini только красивее
формулирует текст и не может на них повлиять.

Чтобы включить LLM-режим, создайте файл `.env` в корне проекта (рядом
с `run.py`):

```
AI_PROVIDER=gemini
GEMINI_API_KEY=...
# необязательно, по умолчанию gemini-flash-latest:
GEMINI_MODEL=gemini-flash-latest
```

Полный шаблон — в `.env.example`.

Ключ используется только для обращения к Gemini API в момент нажатия
«Обновить AI-анализ» и нигде не сохраняется. Если ключ не задан, сеть
недоступна или запрос не удался по любой причине — приложение
автоматически и незаметно для пользователя откатывается на локальный
расчёт (в интерфейсе это видно по бейджу «локально» / «Gemini» рядом
с заголовком блока).

**Важно про сам API-ключ:** относитесь к нему как к паролю. Файл
`.env` уже в `.gitignore` и не попадёт в git по умолчанию — не
вставляйте ключ никуда, кроме этого файла (не в чаты, не в issue, не
в коммиты). Если ключ где-то засветился — считайте его
скомпрометированным и как можно скорее отзовите/пересоздайте в Google
AI Studio / Cloud Console.

### Живая оценка во время переписки

Пока открыт диалог с контактом, добавленным в CRM, справа в панели
контакта показывается «живой» бейдж интереса, который обновляется
сам — без нажатия «Обновить AI-анализ» — по мере того, как приходят
новые сообщения. Это всегда только локальная эвристика (эндпоинт
`/live-score`): она пересчитывается по уже загруженным сообщениям
переписки без обращения к Gemini и без лишних походов в Telegram, поэтому
это можно делать часто, не боясь ни счёта, ни лимитов. Полный анализ
с summary/next_action/черновиком ответа (и, если включён, с Gemini)
по-прежнему запускается вручную кнопкой обновления.

### Прогноз «затухания» переписки (тренд)

И полный анализ, и живая оценка дополнительно показывают тренд —
растёт интерес, стабилен или затухает. Считается так: переписка
делится по времени пополам, для каждой половины отдельно считается
скорость ответа, длина сообщений, доля вопросов и т.д., и половины
сравниваются между собой. Если во второй половине показатели заметно
хуже, чем в первой — это и есть «затухание».

Чтобы не делать выводы на шуме, тренд не показывается, пока в каждой
половине переписки не наберётся хотя бы по 3 ответных сообщения от
контакта — до этого в интерфейсе просто нет чипа с трендом. Это тоже
всегда только локальный расчёт, без Gemini.

### Глубокий AI-отчёт

Отдельная, более тяжёлая фича — кнопка «Сгенерировать отчёт» под обычной
карточкой Contact Intelligence. В отличие от неё, глубокий отчёт **требует
Gemini** (`AI_PROVIDER=gemini` + `GEMINI_API_KEY`) — без него кнопка вернёт
понятную ошибку, локального аналога тут нет: слишком много измерений, чтобы
их честно посчитать без LLM.

Отчёт разбирает всю переписку по многим осям и считается только по явному
запросу (не автоматически, не по таймеру):

- независимая оценка интереса и тренда (своя, не подменяет число из обычной
  карточки — оба остаются рядом как два разных мнения: детерминированное
  локальное и от Gemini);
- инициатива / вложенность / «кто тащит диалог» — доли между вами и
  собеседником;
- вероятность встречи, уместность приглашения прямо сейчас, риск «игнора»;
- индекс вашего собственного давления (слишком частые сообщения, повторные
  вопросы/приглашения) — это про ваше поведение, не про собеседника;
- зелёные/красные флаги, конкретные ошибки в последних сообщениях,
  до 3 рекомендаций именно по этой переписке;
- одно следующее действие и объяснение оценки.

Промпт явно просит модель не фантазировать, не завышать интерес и опираться
на действия, а не на слова — те же принципы, что уже заложены в локальной
эвристике. Результат сохраняется в БД (одним JSON-блобом), поэтому
переживает перезагрузку страницы — обновляется только по нажатию кнопки
(значок ⟳ на уже посчитанном отчёте).

## Работа через прокси/VPN без системного TUN-режима

Если официальный Telegram у вас работает просто с включённым VPN, а эта
панель — нет (зависает при загрузке диалогов или падает с ошибкой вида
`getaddrinfo failed`), причина не в баге, а в разнице между тем, как
подключается официальный клиент Telegram и как подключается эта CRM.

Официальные приложения Telegram умеют сами обходить блокировки: у них
встроены запасные адреса серверов, обфускация трафика и автоматический
подбор рабочего пути подключения. Из-за этого Telegram может "пробиться"
даже через VPN в лёгком режиме (когда VPN туннелирует не весь трафик
системы, а только часть). Backend этой CRM работает через библиотеку
Telethon — обычное MTProto-подключение поверх обычного сетевого стека
ОС, без такой встроенной "живучести". То же самое верно и для запросов к
Gemini API. Поэтому если VPN не в полном системном (TUN) режиме, у
Telegram-приложения путь наружу есть, а у backend'а — нет.

Решение — не переключать VPN в полный TUN-режим, а явно указать backend'у
локальный прокси, который для этого обычно и так поднимает VPN-клиент
(режим "Proxy"/"Local proxy"/"SOCKS" в настройках VPN-приложения — уточните
у вашего провайдера VPN порт и тип, часто это `127.0.0.1:1080` или похожий).
Впишите в `.env`:

```
TG_PROXY_TYPE=socks5
TG_PROXY_HOST=127.0.0.1
TG_PROXY_PORT=1080
```

Если прокси купленный (не от VPN-клиента), обычно у него есть логин и
пароль — просто добавьте их же переменными:

```
TG_PROXY_TYPE=socks5
TG_PROXY_HOST=1.2.3.4
TG_PROXY_PORT=1080
TG_PROXY_USERNAME=ваш_логин
TG_PROXY_PASSWORD=ваш_пароль
```

(если продавец прокси дал только `http`, а не `socks5` — впишите
`TG_PROXY_TYPE=http`, остальное так же).

Этот же прокси автоматически используется и для запросов к Gemini — второй
раз указывать ничего не нужно (если он всё же должен отличаться от
Telegram-прокси, есть отдельная `HTTP_PROXY_URL` — см. `.env.example`).
После правки `.env` перезапустите `run.py`.

## Резервное копирование

Все данные лежат в одном файле: `data/crm.db`. Чтобы сделать бэкап —
просто скопируйте этот файл. Чтобы начать с чистого листа — удалите
его (создастся заново при следующем запуске).

## Дальнейшее расширение

Код разбит на независимые модули, поэтому легко добавить:
- новые поля карточки (`backend/models.py` → `backend/schemas.py` → форма в `frontend/js/contacts.js`);
- новые статусы воронки (`backend/models.py` → `STATUS_LABELS` / `STATUS_ORDER`);
- экспорт в CSV, импорт контактов и т. д. — как новый роутер в `backend/routers/`.
